# Jellyfin Kotlin SDK Album Browse Bug

## Summary

The album undercount is not explained by Gelli's album browse query construction.
The current Java fallback in `QueryUtil.getAlbums(...)` is still justified as a mitigation.

The app-side album request shape is effectively the same before and after the migration:

- `parentId` is set from the selected library.
- `includeItemTypes` is `MusicAlbum`.
- `limit`, `startIndex`, `sortBy`, and `sortOrder` come from the fragment query.
- `userId` is set from the active Jellyfin session.

That means the remaining bug is at the Kotlin SDK layer, not in the album fragment or query translation code in this repo.

## What Was Verified

### Album query setup in Gelli is correct

Album browse starts in `app/src/main/java/com/dkanada/gramophone/fragments/library/AlbumsFragment.java`.

The fragment builds an `ItemQuery` with:

- `includeItemTypes = ["MusicAlbum"]`
- `userId = App.getApiClient().getCurrentUserId()`
- `recursive = true`
- `limit = pageSize`
- `startIndex = adapter item count`
- `parentId = QueryUtil.currentLibrary.getId()`
- sort fields from preferences

The Kotlin migration path translates that into `GetItemsRequest` in `app/src/main/java/com/dkanada/gramophone/util/QueryUtil.kt`.

The translation preserves the important fields:

- `userId`
- `parentId`
- `startIndex`
- `limit`
- `recursive`
- `sortBy`
- `sortOrder`
- `includeItemTypes = [MUSIC_ALBUM]`

No repo-side evidence was found that Gelli drops, rewrites, or mis-scopes the album browse query before it reaches the SDK.

### The artist issue was different

The artist regression had a separate root cause.
The SDK migration changed the endpoint from album artists to generic items.

- Legacy Java path: `GetAlbumArtistsAsync(...)`
- Broken SDK migration path: `itemsApi.getItems(...)` with `includeItemTypes = MUSIC_ARTIST`

That is why the artist issue can be fixed inside Gelli by switching back to the SDK's `artistsApi.getAlbumArtists(...)` route.

The album issue does not have an equivalent app-side endpoint mismatch.

## Concrete SDK-Level Defect Found

While tracing the SDK request path, one protocol bug was confirmed in the Kotlin SDK:

- File: `~/source/jellyfin-sdk-kotlin/jellyfin-api/src/commonMain/kotlin/org/jellyfin/sdk/api/client/util/UrlBuilder.kt`
- Method: `buildUrl(...)`

The SDK serializes collections as repeated query parameters:

```text
sortBy=Album&sortBy=SortName
includeItemTypes=MusicAlbum
fields=Genres&fields=Overview
```

The legacy Java client serializes arrays as a single delimited value using `QueryStringDictionary`:

```text
sortBy=Album,SortName
includeItemTypes=MusicAlbum
fields=Genres,Overview
```

This matters because Jellyfin's OpenAPI contract for many of these fields explicitly says "comma delimited" or "pipe delimited".

Examples in the generated SDK request models:

- `includeItemTypes`: comma delimited
- `sortBy`: comma delimited
- `sortOrder`: comma delimited
- `genreIds`: pipe delimited
- `officialRatings`: pipe delimited
- `tags`: pipe delimited

The current Kotlin SDK URL builder ignores those delimiter requirements and instead emits one query parameter per collection element.

## Why This Does Not Fully Explain the Album Bug Yet

This serialization bug is real, but it is not sufficient by itself to fully explain the default album browse regression in Gelli.

For the default album browse request, the most important collection fields are usually single-element:

- `includeItemTypes = [MusicAlbum]`
- `sortBy = [one sort field]`
- `sortOrder = [one sort order]`

Single-element collections serialize acceptably either way.

Because of that, the album undercount still needs deeper SDK-side debugging.

## Current Best Hypothesis

The remaining album bug is most likely one of these:

1. Another SDK request serialization mismatch that is not visible from the repo layer.
2. A transport or query-building issue in the SDK path for `/Items`.
3. A Jellyfin server behavior difference triggered by the SDK wire format even when the logical request fields appear equivalent.

At this point, the app code in Gelli is not the highest-value place to keep guessing.
The next debugging pass should happen in `~/source/jellyfin-sdk-kotlin`.

## Recommended SDK Debugging Next

Do this in the SDK repo instead of patching Gelli:

1. Log the exact final album browse URL generated by the Kotlin SDK.
2. Capture the exact legacy Java URL for the same album browse request.
3. Diff the two URLs byte-for-byte.
4. Fix collection serialization in `UrlBuilder.kt` so it respects the API's delimiter rules.
5. Re-test album counts after that change before investigating deeper request semantics.

## Current Gelli Decision

Keep the album path on the legacy Java fallback for now.
Fix the artist path in Gelli by using the Kotlin SDK's correct album-artists endpoint.
